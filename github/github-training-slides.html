PART 1

Git と GitHub って何？

1Git = ファイルのセーブシステム

ファイルの変更履歴を全部記録してくれる仕組み

🎮 ゲームのセーブと同じ。いつでも過去に戻れる
📝 いつ・誰が・何を変えたか全部残る
🔒 上書きしても元に戻せるから安心

2GitHub = チームで共有するクラウド

Git の記録をインターネット上に置いて、チームでアクセスできるようにする場所

Git
記録する仕組み
自分のPCの中で動く
→
GitHub
共有する場所
クラウド上に保管

💡 Google ドライブを想像するとわかりやすい。
　 Git = 手元のファイル操作、GitHub = ドライブにアップして共有

3覚えておく用語

用語 意味 たとえ
ローカル 自分のPC 手元のノート
リモート GitHub 上（クラウド） クラウドに保存したノート
リポジトリ 1プロジェクト = 1フォルダ 案件ごとのフォルダ
┌─────────────┐ ┌─────────────┐ │ ローカル │ ←────→ │ リモート │ │ (自分のPC) │ │ (GitHub) │ └─────────────┘ └─────────────┘
PART 2

基本操作を覚える

1全体の流れ

あなたのPC（ローカル） GitHub（リモート） ┌──────────────────┐ ┌──────────────────┐ │ │ push → │ │ │ ファイルを編集 │ ───────→ │ チームに共有 │ │ ↓ │ │ │ │ add → commit │ ← pull │ │ │ （選ぶ→セーブ） │ ←─────── │ 他の人の変更 │ └──────────────────┘ └──────────────────┘

24つの操作

git add
「これをセーブに含めて」
レジカゴに商品を入れるイメージ
git commit
「セーブ確定！」
レジで会計するイメージ（PC内）
git push
「GitHub にアップ」
ローカル → リモートへ送る
git pull
「最新をダウンロード」
リモート → ローカルに取り込む

📌 順番: add → commit → push（選ぶ → セーブ → アップ）

3実際のコマンド

# ① 変更したファイルを全部選ぶ git add . # ② セーブ確定（メッセージをつけて） git commit -m "トップページのデザインを変更" # ③ GitHub にアップ git push # ④ 他の人の変更を取り込む git pull
PART 3

ブランチとマージ

チーム作業で一番大事な概念

1ブランチ = 下書きスペース

main ブランチ
📗 清書ノート — 完成品だけが入る = 本番
作業ブランチ
📝 下書きノート — 自由に作業できる = 安全な実験場

✅ ブランチで何をやっても、main には一切影響しない

2なぜブランチが必要？

例: 3人でWebサイトを開発する場合

main ─────────────────────────────────→ 本番（直接変更しない） │ ├── feature/top-page （上田の作業場） │ ├── feature/contact-form （Aさんの作業場） │ └── feature/about-page （Bさんの作業場） → お互いの作業が干渉しません

💡 ブランチなしだと、同じファイルを同時に編集してどちらかの変更が消える危険がある

3ブランチの命名ルール

接頭辞 用途 例
feature/ 新機能を作るとき feature/contact-form
fix/ バグを直すとき fix/phone-number-bug
update/ 既存の機能を改善するとき update/header-design
docs/ ドキュメントを変えるとき docs/readme

📌 名前を見ただけで「何の作業か」がわかるのがポイント

4マージ = 下書きを本番に合流

main ──────●─────────────────●──────→ \ ↑ \ / マージ！ \ / ●───●───●─┘ ブランチで作業

作業が完成したら、ブランチの内容を main に合流（マージ）させる。
ただし、チーム開発では直接マージせず PR を使う（次のパートで説明）

5コンフリクト = 「どっち使う？」

2人が同じファイルの同じ行を変更すると発生

// Git がこんな印をつけてくれる <<<<<<< HEAD <h1>新しいデザイン</h1> ← 自分の変更 ======= <h1>リニューアル版</h1> ← 相手の変更 >>>>>>> feature/other

✅ 壊れているわけではありません。Git が「確認してください」と伝えているだけです。
　 残したい方を選んで、マーカーを消せば解決。

PART 4

チーム開発の進め方

1GitHub のプラン比較

チーム開発に必要な機能はプランによって変わる

機能 Free（無料） Team（$4/人/月）
リポジトリ 無制限（公開・非公開） 無制限（公開・非公開）
ブランチ保護 基本的なルールのみ 高度なルール設定
PRの必須レビュー 設定不可 承認なしでマージ禁止にできる
コードオーナー なし 自動でレビュー担当を割り当て
下書きPR なし 作業途中でPRを共有できる
Actions（自動化） 2,000分/月 3,000分/月
ストレージ 500 MB 2 GB

2Team プランで何が変わる？

🔒
main ブランチを守れる
レビューなしでのマージをシステムで禁止できる
👥
レビュー担当の自動割り当て
PRを作ると自動で担当者に通知が飛ぶ
📝
下書きPR
「作業途中ですが確認お願いします」を気軽に共有
💰
コスト感
4人で $16/月（約¥2,400）
1人あたり月600円程度

💡 まずは Free プランで始めて、チーム開発に慣れたら Team への移行を検討。
　 Opus.net の Organization「team-opus」は作成済み！

3Pull Request（PR）= 承認フロー

マージする前に、チームメンバーに確認・承認してもらう仕組み

直接マージ（危険）
作業完了 → 即マージ
誰も確認してない
PR を使う（安全）
作業完了 → PR → 確認 → 承認 → マージ

💡 書類の稟議・承認フローと同じ

4Issue = タスクチケット

GitHub 上の「やることリスト」。バグ報告や機能追加の依頼に使う。

🐛
バグ報告
「ここが壊れてます」
✨
機能追加
「こんな機能が欲しい」
📋
タスク管理
「これやる必要あるよね」

💡 PR に Closes #3 と書くと、マージ時に Issue #3 が自動で完了になる

6覚えるコマンドは6つだけ

# ブランチを作って移動（作業開始） git checkout -b feature/○○ # 変更したファイルを全部選ぶ git add . # セーブする（メッセージつき） git commit -m "やったことを書く" # GitHub にアップ git push -u origin feature/○○ # main に戻る git checkout main # 最新を取得 git pull

7よくあるトラブル

push できない！
誰かが先に push した
→ git pull してから再度 push
コンフリクト！
同じ行を2人が変更した
→ 残す方を選んでマーカーを消す
今どのブランチ？
git branch で確認
→ * がついてるのが今の場所
間違えて main で作業した
git stash → ブランチ作成
→ git stash pop で変更を戻す

8チームのルール

やってはいけないこと

  • main に直接 push しない（必ずブランチ → PR → マージ）
  • 他の人のブランチを勝手に変更しない
  • git push --force は使わない（履歴が壊れる）

守ること

  • 作業は必ずブランチで行う
  • main への反映は必ず PR 経由
  • PR にはわかりやすいタイトルと説明を書く
  • 困ったら git status で状況を確認

まとめ

この4つだけ覚えて帰ってください

add → commit → push
基本の流れ
ブランチで作業
main は直接変更しない
PR でレビュー
確認してからマージ
困ったら git status
壊れることはない
